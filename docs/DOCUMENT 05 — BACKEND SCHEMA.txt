7. SECURITY RULES OVERVIEW

users

Customer can read and update only own profile.

Admin can read and update all users.

Technician can read own profile.

---

devices

Customer can read and write only own devices.

Admin can read all devices.

---

repair_orders

Customer can create repair orders.

Customer can read only own repair orders.

Technician can read only assigned repair orders.

Technician can update limited status fields only.

Admin can read and write all repair orders.

---

products

Anyone can read active products.

Admin can create, update, and delete products.

---

store_orders

Customer can read only own orders.

Admin can read and update all orders.

---

payments

Customer can read own payments.

Payment creation and verification must happen through Cloud Functions only.

Admin can read all payments.

---

support_tickets

Customer can create and read own tickets.

Admin can read and update all tickets.

---

notifications

Customer can read own notifications.

System/Admin can create notifications.

---

8. SENSITIVE DATA RULES

Never store:

Raw passwords

Card numbers

CVV

Bank details

Razorpay secret key

WhatsApp API token

Firebase service account in frontend

---

Store only:

Payment IDs

Order IDs

Payment status

Webhook verification result

9. CLOUD FUNCTIONS

createRazorpayOrder

Creates Razorpay order securely.

Used by:

Repair payment

Store checkout

Membership payment

---

verifyRazorpayWebhook

Verifies Razorpay webhook signature.

Updates payment status.

Updates order payment status.

---

assignTechnician

Assigns technician to repair order.

Creates repair timeline update.

Sends notification.

---

updateRepairStatus

Updates repair order status.

Creates repair_status_history entry.

Sends push notification.

Sends WhatsApp notification.

---

sendWhatsAppNotification

Sends booking, payment, and repair status updates.

---

createNotification

Creates in-app notification.

---

onOrderCreated

Triggers after repair or store order creation.

Sends confirmation notification.

11. BACKEND RULES

Every user must have a users/{uid} profile.

Every repair order must belong to a user.

Every store order must belong to a user.

Every payment must be verified server-side.

Every repair status update must create a history record.

Every technician update must be tied to assigned order only.

Every admin action should be logged in future audit logs.

13. FINAL BACKEND PRINCIPLE

The backend must protect the business from:

Fake payments

Unauthorized admin access

Technician misuse

Customer data leakage

Product stock errors

Repair status manipulation

Unverified order completion

FixBee backend must be built like a real company system from day one.
