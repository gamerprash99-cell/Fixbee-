5. CORE USER JOURNEY 1 — BOOK A REPAIR

Goal: User wants to repair a phone.

Flow:

Home
→ Tap Repair Device
→ Select Brand
→ Select Model
→ Select Issue
→ View Price Estimate
→ Choose Service Type
→ Select Address
→ Schedule Pickup
→ Review Booking
→ Payment
→ Booking Success
→ Repair Tracking

Repair Tracking Timeline:

1. Request Created
2. Technician Assigned
3. Device Picked Up
4. Repair Started
5. Quality Check
6. Ready For Delivery
7. Delivered

Redirect Logic:

After successful booking → Repair Tracking Screen
After payment failed → Payment Failed Screen
After cancellation → Orders Screen

15. REDIRECT RULES

After signup → Create Profile
After profile creation → Add First Device
After first device added → Home
After login → Home
After logout → Login
After repair booking → Repair Tracking
After store checkout → Store Order Tracking
After failed payment → Payment Failed Screen
After support ticket submission → Ticket Detail
After admin login → Admin Dashboard
After technician login → Technician Dashboard
If non-admin opens admin → Access Denied
If non-technician opens technician panel → Access Denied

---

16. GLOBAL APP RULES

Customer should never see admin screens.

Customer should never see technician screens.

Technician should only see assigned jobs.

Admin can access full platform.

Every protected screen requires authentication.

Every order must have a status.

Every payment must have a verified backend record.

Every repair status update must create a timeline entry.

Every product order must create an order record.

Every major action should create a notification.

---

17. FINAL EXPERIENCE GOAL

FixBee should feel like one connected product.

The user should move smoothly from:

Device
→ Repair
→ Tracking
→ Store
→ Protection
→ Support

The experience should feel premium, trustworthy, fast, and simple.
